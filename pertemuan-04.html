Pertemuan-04 — Test Scenario, Test Case, and Bug Report

Mendokumentasikan pengujian perangkat lunak secara terstruktur dan profesional

1. Pengantar

Dalam pengujian perangkat lunak, dokumentasi yang baik sangat penting untuk memastikan seluruh fitur telah diuji secara sistematis. Dua dokumen utama dalam proses tersebut adalah Test Scenario dan Test Case, sementara Bug Report digunakan untuk mencatat kesalahan yang ditemukan selama pengujian.

2. Pengertian Test Scenario dan Test Case

  • Test Scenario adalah gambaran umum dari suatu proses atau alur pengujian untuk memastikan fungsi sistem bekerja sesuai kebutuhan. Scenario menjawab pertanyaan "Apa yang akan diuji?".
  • Test Case adalah langkah-langkah rinci yang dilakukan untuk menguji suatu skenario. Test case menjawab pertanyaan "Bagaimana pengujiannya dilakukan?".

Tujuan Pembuatan Test Scenario dan Test Case

  • Menjamin setiap fungsi sistem diuji dengan cara yang konsisten dan dapat diulang.
  • Memastikan pengujian mencakup semua kebutuhan fungsional perangkat lunak.
  • Mempermudah pelacakan bug dan validasi hasil perbaikan.

3. Template Sederhana Test Scenario & Test Case

Berikut contoh format sederhana yang dapat digunakan untuk mendokumentasikan pengujian:

Contoh Template Test Scenario

ID Scenario Deskripsi Tujuan Pengujian
TS-001 Menghitung nilai BMI berdasarkan input berat dan tinggi Memastikan sistem dapat menghitung BMI dengan benar
TS-002 Menampilkan kategori hasil BMI Memastikan hasil BMI sesuai klasifikasi WHO

Contoh Template Test Case

ID Case ID Scenario Langkah Pengujian Data Uji Hasil yang Diharapkan Hasil Aktual Status
TC-001 TS-001 Masukkan berat = 60 kg dan tinggi = 165 cm, lalu klik “Hitung” 60, 165 Menampilkan hasil BMI = 22.04 22.04 Lulus
TC-002 TS-002 Input berat = 85 kg dan tinggi = 160 cm, lalu klik “Hitung” 85, 160 Menampilkan kategori = “Obese” Obese Lulus
TC-003 TS-001 Masukkan nilai tinggi kosong dan tekan “Hitung” 60, (kosong) Menampilkan pesan error “Tinggi badan tidak boleh kosong” Tidak menampilkan pesan Gagal

4. Pengertian Bug Report

Bug Report adalah dokumen yang berisi deskripsi terperinci tentang kesalahan atau cacat (bug) yang ditemukan selama pengujian. Tujuannya agar tim pengembang dapat mereproduksi dan memperbaiki bug dengan cepat.

Informasi yang Harus Ada dalam Bug Report

  • ID unik untuk setiap bug.
  • Judul atau ringkasan singkat bug.
  • Langkah-langkah untuk mereproduksi bug.
  • Hasil aktual dan hasil yang diharapkan.
  • Tingkat keparahan (severity) dan prioritas.
  • Lingkungan pengujian (OS, browser, versi aplikasi).
  • Status bug (baru, diperbaiki, diverifikasi, ditutup).

Contoh Format Bug Report

ID Bug Judul Deskripsi Langkah Reproduksi Hasil Aktual Hasil Diharapkan Severity Status
BUG-001 Pesan error tidak muncul saat input kosong Pengguna tidak menerima peringatan ketika kolom tinggi dibiarkan kosong. Masukkan berat = 60, biarkan tinggi kosong, tekan “Hitung”. Aplikasi tetap memproses tanpa pesan error. Sistem menampilkan pesan “Tinggi badan tidak boleh kosong”. Medium Open

5. Kesimpulan

Test Scenario dan Test Case merupakan fondasi dokumentasi dalam proses pengujian perangkat lunak. Dengan menulis skenario dan kasus uji yang baik, tim dapat menguji sistem secara menyeluruh dan mengurangi risiko kesalahan. Sementara itu, Bug Report berfungsi sebagai alat komunikasi utama antara penguji dan pengembang dalam memperbaiki kualitas aplikasi.
